# Future Work

This document collects open follow-ups on the mechanistic-interpretability study
of HRM. Items are loosely ordered by expected payoff for a publication-quality
extension of the current results.

## A. Tighter, larger-N replications

- **E1 (single-step ablation)**: re-run with N ≥ 5,000 puzzles per step, ablating
  one ACT step at a time independently. Add the symmetric **E1b** that ablates
  z<sub>L</sub> (currently only z<sub>H</sub> is covered).
- **E2 (freeze-z<sub>H</sub>)**: lift to N ≥ 1,000, sweep all freeze steps
  0–15, and add a freeze-z<sub>L</sub> control.
- **E5 (time-shift patching)**: complete the full source × target step transfer
  matrix at N ≥ 500 with random-direction controls.
- **E9 (directed ablation)**: re-run at N ≥ 500 with proper statistical tests
  (paired Wilcoxon, bootstrap CI, effect sizes), and try multi-direction
  ablation (project out the top-k probe directions jointly).

For each experiment add 95 % bootstrap CIs and Cohen's d in the plotting
scripts (`scripts/plotting/*`).

## B. Beyond linear probes — better feature discovery

E9 demonstrated that linear probe directions are *readout* features, not
*computational* features. Promising next steps:

- **Sparse Autoencoders on z<sub>H</sub> (E10)**: dictionary sizes 1024–4096
  with L1 sparsity sweep. Compare causal specificity of individual SAE
  features against probe directions on the same constraint targets. Initial
  results live under `results/sae_study/`; gaps include per-step / per-head
  feature dictionaries and joint top-k feature ablations.
- **Cross-coder / transcoder analysis** between z<sub>H</sub> at successive
  steps to find features that *transmit* information across iterations.
- **Activation steering**: add (rather than subtract) probe / SAE directions to
  z<sub>H</sub> mid-computation and measure the violation deltas.

## C. Generalisation to other domains

- Re-run the full E1/E2/E5/E8/E9/E10 suite on **Maze 30×30 hard** (started
  under `results/maze/`) and on **ARC-AGI-2**. Tasks with longer horizons may
  expose computational features that Sudoku does not require.
- **Difficulty-stratified analysis** (currently in `scripts/analysis/`):
  separate easy vs. hard puzzles and re-compute every interpretability metric.
  Hypothesis: hard puzzles use more of the constant-norm rotation budget of
  z<sub>H</sub>.

## D. Architectural follow-ups (HRM v2)

`models/hrm_v2/` adds three structural inductive biases:
constraint-aware sparse attention, an explicit constraint-satisfaction head,
and a small graph-style refinement layer. Open questions:

- Train HRM v2 to convergence on Sudoku-Extreme (full) and rerun the entire
  MI suite. Does the explicit constraint head produce *causally specific* probe
  directions (i.e., a positive E9 result)?
- Ablate each of the three HRM v2 components individually to attribute any
  improvement.

## E. Scaling and statistical rigour

- A single shared puzzle set (with fixed seed) across every experiment so that
  E1/E2/E5/E8/E9/E10 numbers can be compared cell-by-cell.
- Reporting random seeds, hardware, and full Hydra configs alongside every
  results JSON (write a small `_meta.json` per experiment).
- A unified plot style (ColorBrewer + consistent axis labels) for the
  publication figures regenerated by `scripts/plotting/generate_paper_figures.py`.

## F. Conceptual / writing

- Frame the z<sub>H</sub> finding as evidence (or counter-evidence) for an
  **emergent world-representation** of the Sudoku board, in dialogue with the
  Othello-GPT line of work.
- Compare against the **structured-guesser** characterisation of Ren et al.
  (2026) and clarify where our causal interventions agree or disagree.
- Add an explicit **negative-results** section centred on E9 ("readout ≠
  computation") so reviewers do not mistake it for a failed experiment.

## G. Engineering / repository

- Tighten the configs so `pretrain.py` and `evaluate.py` work without setting
  any environment variables (today the bash wrappers set `OMP_NUM_THREADS`,
  `DISABLE_COMPILE`, etc.).
- Replace the SLURM headers in `scripts/bash/train_*.sh` with a thin wrapper
  that picks SLURM vs. local execution based on `command -v sbatch`.
- Provide a `make repro-sudoku` / `make repro-maze` target that runs the small
  subset of experiments needed to regenerate every committed paper figure.
